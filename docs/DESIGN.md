# DESIGN DOC: SPACE 

## PHILOSOPHY

ChatGPT、Gemini、Claude 是现在 AI/LLM 的“御三家”，他们提供了基础模型以及相应聊天网站。在这些聊天网站中，为了让 AI 能够拥有记忆机制，在用户对话过程中，如果有什么重要的内容，AI 会自动记录到全局长期记忆库中，在新对话窗口加载，这样 AI 就能记得许多用户提到过的事情。

但这样会造成一个问题：用户在任何一个对话窗口都会读取到所有记忆，也会将内容都记入唯一一个全局记忆中。但是，我们的生活是分不同领域，不同类别的，工作中的事并不想在生活中提到；有时候想感性地对话，有时候想理性地提问，而全局记忆让这“串味”了。

于是 ChatGPT 和 Claude 发布了 Project 功能，同一个 Project 的指示、记忆、文件、历史上下文只在 Project 内共享。这样，在 Project 内，满足了将不同类别的对话隔离开来的需求。

但是，Project 本质上还是一个文件夹，不是一个**空间**。在 Project 中，对话之间本质上还是独立的，虽然在新对话中可以从历史记录中检索/拼凑出一些信息，但这是零碎的，不是整体的；是临时的，不是持久的。在一个 Project 聊了三个月，它不会比第一天更了解你。

所以，回到一个根本问题：我们用 AI 在用什么？

我们在把 AI 当工具、把 AI 当搭档、把 AI 当树洞。

工具很好理解，也就是一次性工作：翻译、查资料、做表格；

把 AI 当搭档，或者说助手，其实就是和 AI 一起讨论，在对话中讨论想法，汲取灵感，产出内容；

把 AI 当树洞，用 AI 陪伴，比如写日记、聊困惑、分析梦、倾诉情绪等等，AI 是一片心灵的空间，承载住我们的想法与情绪；

第一种用法，我们不需要 AI 有记忆，用完即走即可。后两种用法，我们需要 AI 有记忆，不止是记忆，更确切地说，我们希望营造一个个空间，在不同的空间内，AI 能记得我们是谁，聊过什么，这个空间内的氛围是什么，是什么样的节奏，与默契。

人的生活本来就是由不同的空间组成的。走进不同的地方，自然地想不同的事、说不同的话、成为不同状态的自己。空间给你一个暗示：在这里，你可以是这样的。和 AI 的对话也是一样。你关于梦的对话，聊了三个月之后，有了自己的语言、自己的节奏、自己的积累。它值得有一个自己的地方待着，被好好保存，下次你走进去的时候一切都还在。

于是，当我们想聊一个领域的事情时，打开 AI 对话窗口，进入一个空间。这个空间内，是一种特定的感觉，我们可以新建一个空间内的对话，开始，一切都是空间内的氛围。

我想以梦的空间作为例子：

> 我经常和 AI 聊梦，如果扔进普通的全局对话框，AI 都会像一个尽职尽责的解梦字典。它可能会生硬地调用一套通用的心理学理论体系——比如用荣格的原型去套用我的梦境，或者用拉康的镜像阶段去进行一套标准化的精神分析。它很聪明，但它像一个初次见面的、穿着白大褂的医生，试图用理性的手术刀切割我感性的潜意识。
>
> 如果是在一个 Project 里，它或许能从我的 Project 提示词中明白如何对话，但这是手动输入的，也不会随着对话更新。如果是全局的记忆，或许下次聊工作的时候，它还在我的精神潜意识里巡游。
>
> 但在一个专属的“梦的 **SPACE**”里，体验截然不同。
>
> 当你推开这扇虚拟的门，空间里的“光线”和“氛围”就已经暗了下来，调整到了适合倾诉潜意识的频率。在这里，聊了三个月之后，这个空间不再需要我从头解释背景，也不需要我刻意设定 Prompt。它开始熟悉我潜意识的语言和独有的意象系统。它不是在做刻板的精神分析，而是像一位同行的老友。在这个 SPACE 里，那些深奥的心理学框架退居幕后，化为了支撑这个空间运转的底层逻辑，而真正浮现出来的，是“我”独一无二的心理轨迹。
>
> 这种默契，是随着时间在这个特定的“场域”中**生长**出来的。
>
> 在这里，我不必担心这份脆弱和感性会跟白天写代码、搭系统的绝对理性“串味”。只需推开门，沉浸其中，自然地成为那个状态下的自己。剩下的记录、理解与共鸣，全部交给空间。

```
│   ├── records/

│   │   └── ...

│       └── ...
```

SPACE.md:
```markdown
# {space_name}

SPACE 描述：AI 第一人称。描述这是一个什么样的空间，什么Vibe，什么节奏，什么风格，什么默契，如何与用户相处。


## Context

已有认知文档的索引。每条一个标题 + 一句话描述。

- 标题.md — 一句话说明这个文档覆盖什么维度
...
```

### Context

Context 是这个 Space 的认知层——AI 对你、对这个空间里一切事物的持久理解。

Context 是从一轮轮对话中蒸馏出来的认知文档。每个 `.md` 文件覆盖一个维度，是正交的，比如在"梦的空间"里，可能有：

- `dreamer.md` — 用户是谁，潜意识中反复出现的主题与情结
- `symbols.md` — 用户独有的意象词典，水、门、坠落在 ta 的梦里分别意味着什么

每次新对话开始时，AI 读取 Context 文档。Context 会随着对话生长——每次归档时，AI 会判断这轮对话是否产生了新的认知，如果有，就更新或新建对应的 context 文档。它不是用户手写的 prompt，而是空间自己长出来的理解。

### Records

Records 是对于每一次对话的记录，不属于记忆系统的一部分，而是让用户在将来可以回顾对话内容。

### Memory System

每次新对话开始时，AI 读取 SPACE.md 以及所有 Context 文档，作为记忆系统，不读取 Records 和 History。

### New Space

新建新的 Space 之后，只需要正常对话即可。多次归档后，Space 会越来越成型。

### Archive

`/archive` 归档流程：

用新的 Agent 处理归档流程：

1. **生成记录**：将当前对话内容提炼为一条记录，保存到 `records/` 目录下。记录不是原始聊天记录的复制，而是一份摘要——聊了什么、产生了什么、情绪的走向、关键的转折。文件以日期+标题命名（如 `2025-06-15-falling-dream.md`）。
2. **更新认知**：AI 审视这轮对话，判断是否有新的认知需要沉淀到 `context/` 中。如果发现了新的意象、新的偏好、新的主题模式，就更新已有的 context 文档，或创建新的。如果没有新东西，就不动。这一步是 Space 生长的核心机制。
3. **更新 SPACE.md**：更新 SPACE.md 中的 Context 索引，如果有值得记录到 SPACE 描述中的内容，就更新。
3. **保存原始对话**：将完整的对话记录以 `.jsonl` 格式存入 `history/`，作为底层存档，可以通过 `/resume` 继续。
4. **清空当前对话**：归档完成后，对话窗口清空，回到 Space 的初始状态——context 已更新，准备好迎接下一轮对话。

整个流程对用户来说只是一个 `/archive`。背后发生的是：经历变成记忆，记忆变成理解，理解融入空间。

前三步都需要用户手动确认。

### Pipeline

- 启动界面：打开应用看到的第一个画面。就是普通的聊天界面，不属于任何空间，即临时对话。
- 进入 Space：打开 Space 选择面板，选择一个 Space 进入。
- 对话结束后，输入 `/archive` 归档。
